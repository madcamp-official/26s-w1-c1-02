## 기획안

> 프로젝트 주제, 목적, 핵심 기능, 예상 사용자, 팀원별 역할 등 정리

- **주제**: 미니게임천국 — 자모 조합, 다른 그림 찾기, 끝말잇기 세 종류의 캐주얼 미니게임을 한곳에서 즐기는 웹 서비스
- **목적**: 설치나 진입장벽 없이 브라우저에서 바로 즐길 수 있는 킬링타임용 미니게임을 제공하고, 혼자 하는 레벨 도전과 친구와 겨루는 실시간 대전을 함께 지원
- **핵심 기능**:
  - 로그인 없이도 즉시 플레이 가능한 **손님 모드**, 로그인 시 게임별 레벨 진행도 서버 저장
  - 레벨(1~20)이 오를수록 난이도가 상승하는 **싱글플레이 3종**(자음·모음 조합, 다른 그림 찾기, 끝말잇기)
  - 6자리 코드로 방을 만들고 참가하는 **실시간 대기실**(로비/방 채팅, 준비 상태, 방장 위임)
  - 같은 방에서 즉시 시작하는 **실시간 대전 3종**(자모 레이스, 다른 그림 찾기 대전, 숫자야구 1대1)
- **예상 사용자**: 짧은 시간에 간단히 즐길 놀거리를 찾는 10대~30대, 친구와 함께 실시간으로 겨루고 싶은 사용자
- **팀원별 역할**: 아래 [팀원](#팀원) 표 참고

## 기능 명세서

> 구현할 기능을 사용자 관점에서 정리하고, 필수 기능과 선택 기능을 구분

### 필수 기능

**계정 · 프로필**
- 아이디/이메일/비밀번호/닉네임으로 회원가입 및 로그인
- 소셜 로그인(카카오 · 구글 · 네이버) — 최초 로그인 시 닉네임 설정 화면으로 유도
- 첫 화면에서 **손님으로 계속하기** — 로그인 없이 바로 입장. 멀티플레이는 가능하지만 레벨 진행도 저장은 안 됨(브라우저 임시 저장만)

**싱글플레이 미니게임 (레벨에 따른 난이도 상승)**
- **자음·모음 조합하기**: 셔플된 자모를 조합해 단어 맞히기. 난이도별 음절 수 증가, 정답 시 자모 수·풀이 속도·난이도 기반 점수 산정
- **다른 그림 찾기**: 두 그림에서 다른 부분 찾기. 레벨이 오를수록 격자 크기 증가
- **끝말잇기**: 서버가 매판 랜덤 시작 단어 지급, 두음법칙 반영 체인 검증, 특정 턴마다 글자 수 제한이 붙는 보스 턴, 위험 상황 보너스 점수, 목숨을 소모하는 초성 힌트. 레벨(1~20)은 목표 체인 개수 달성 기준으로 클리어
- 세 게임 모두 유저별·게임별로 독립된 서버 진행도 저장(로그인 유저 대상) — 깬 레벨까지 다음 접속 시 이어서 도전 가능

**실시간 대기실(멀티플레이 로비)**
- 6자리 코드로 방 생성/참가, 비밀번호를 건 비밀방 개설
- 방 인원 2~8명(1대1 전용 게임은 2명 고정), 방장이 방 나가면 다음 인원에게 방장 자동 위임
- 방장 권한: 게임 모드 변경, 난이도·문제 수 설정, 게임 시작/종료
- 참가자 준비 상태 토글 — 방장 제외 전원 준비돼야 시작 가능
- 로비 전체 채팅과 방별 채팅, 실시간 접속자 목록 표시

**개인전 실시간 배틀**
- **자음·모음 조합 레이스**: 같은 문제를 동시에 풀어 먼저 맞힌 순서·속도로 점수 산정, 제한시간 2/3 경과 시 힌트 공개, 난이도 4단계(쉬움/보통/어려움/세종대왕)
- **다른 그림 찾기 대전**: 두 격자에서 다른 칸을 먼저 클릭하는 사람이 득점, 오답 시 쿨다운으로 연타 방지, 난이도 4단계(격자 5×5~11×11, 최고 난이도 인간프린터)
- **숫자야구 1대1**: 정원 2명 고정, 서로 비밀 숫자를 정한 뒤 번갈아 추측해 스트라이크/볼/아웃 판정, 먼저 정답을 맞히면 승리(동시 홈런 시 무승부), 중도 퇴장 시 남은 플레이어 부전승

### 선택 기능

- 닉네임 변경, 아바타(이모지) 변경 — **로그인 후에만 이용 가능**

## 배포 결과물

> 접속 가능한 링크, 실행 방법, 주요 구현 내용

- **서비스 URL:** https://minigameheaven-v1.madcamp-kaist.org
- **주요 구현 내용:** [기능 명세서](#기능-명세서) 참고 — 계정/소셜 로그인, 레벨제 싱글 미니게임 3종, 6자리 코드 기반 실시간 대기실, 실시간 대전 3종

### 실행 방법 (로컬 개발 환경)

백엔드(Node.js + Socket.IO, Docker Compose)와 정적 프론트엔드(nginx 또는 임의 정적 서버)로 구성됩니다.

```bash
# 1. 저장소 클론
git clone https://github.com/madcamp-official/26s-w1-c1-02.git
cd 26s-w1-c1-02

# 2. 백엔드 환경변수 설정 (.env.example 복사 후 JWT_SECRET 채우기, 나머지는 선택)
cd backend
cp .env.example .env

# 3. DB + 웹소켓 서버 기동 (docker-compose.yml이 postgres, ws-server 컨테이너를 함께 올림)
docker compose up -d --build

# 4. 프론트엔드는 정적 파일이므로 별도 서버로 서빙
cd ../frontend
python3 -m http.server 3000
# → http://localhost:3000 접속 (API 요청은 백엔드 8080 포트로 프록시 설정 필요,
#   실서비스에서는 backend/nginx/minigameheaven.conf 참고)
```

실서비스 배포는 `backend/scripts/deploy.sh`가 프론트엔드 정적 파일을 nginx 웹 루트로 복사하고 백엔드 컨테이너를 재빌드하는 방식으로 자동화되어 있습니다(캐시 버스팅 빌드 ID 자동 갱신 포함).

```bash
sudo ./backend/scripts/deploy.sh
```

## 회고 문서

> 개발 과정에서의 어려움, 해결 방법, 역할 분담, 다음에 개선할 점 (KPT 방법론 참고)

### Keep

- 미니게임 3종 + 실시간 대전이라는 아이디어 자체가 좋았고, 실제로 구현까지 이어져 만족스러웠음
- 개발 초반부터 서버를 올려두고 실제 배포 환경에서 테스트하며 진행한 덕분에, "로컬에서는 되는데 배포하면 안 되는" 이슈를 초기에 발견하고 대응할 수 있었음

### Problem

- DB 스키마를 개발 초기에 확정하고 가지 않아서, 기능을 추가할 때마다 테이블/컬럼 구조를 바꾸는 일이 반복됨 — 마이그레이션 부담과 코드 수정 범위가 커짐
- 처음에 계정 DB(MongoDB)와 게임 DB(PostgreSQL)를 분리했다가 결국 PostgreSQL 단일 DB로 통합한 것도 초기 스키마 설계 부족이 원인 중 하나

### Try

- 자모 조합/다른 그림 찾기/끝말잇기 외에 추가 미니게임 종류를 늘려서 콘텐츠 다양성 확보
- 다음 프로젝트에서는 ERD/스키마를 개발 착수 전에 먼저 확정하고 시작하기
